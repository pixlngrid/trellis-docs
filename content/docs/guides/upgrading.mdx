Starting with v1.6.1, {vars.productName} includes a CLI upgrade command that updates framework files and dependencies in an existing project. For older versions, follow the manual steps below.

## Upgrade with the CLI

Run from your project root:

```bash title="Terminal"
npx create-trellis-docs@latest upgrade
```

This will:
- Copy updated framework files (components, lib, scripts, app routes)
- Merge new dependencies and scripts into your `package.json`
- Skip your user files (`config/`, `content/`, `public/`, `components/brand/`)

Preview what would change without modifying anything:

```bash title="Terminal"
npx create-trellis-docs@latest upgrade --dry-run
```

After upgrading, review the diff, install, and build:

```bash title="Terminal"
git diff
npm install
npm run build
```

:::tip
The CLI upgrade does not modify your `config/site.ts`. Check the version-specific sections below for any new config options you should add manually.
:::

---

## Manual upgrade steps

If you prefer to upgrade manually, or are upgrading from a version older than v1.6.1:

1. **Check the release notes** — Read the [release notes](/release-notes/) for all versions between your current version and the target
2. **Follow the version-specific steps below** — Each section lists the exact files and dependencies to update
3. **Install and build** — Run `npm install` then `npm run build` to verify everything works

:::tip
To see the full latest template, scaffold a fresh project in a temporary directory with `npx create-trellis-docs@latest temp-docs` and diff it against your project. This is a reference copy for comparison — don't run the scaffolder inside your existing project.
:::

## v1.6.1

Security fix: upgrade `next-mdx-remote` from v5 to v6.

### Step 1: Update dependency

```bash title="Terminal"
npm install next-mdx-remote@^6.0.0
```

This is a drop-in replacement — the `MDXRemote` API is unchanged for server components.

### Step 2: Verify the build

```bash title="Terminal"
npm run build
```

No code changes are required. If you use `MDXRemote` in any custom components, the import path and props remain the same.

**`app/blog/page.tsx`** and **`app/blog/[slug]/page.tsx`** — Replace with the latest template versions. These now import layout components instead of rendering inline.

**`app/release-notes/page.tsx`** and **`app/release-notes/[slug]/page.tsx`** — Same as above.

**`components/docs/mdx/code-block.tsx`** — Add title extraction from meta string:

```ts title="code-block.tsx" {3-4,10}
  const code = children.trim()

  // Extract title="..." from meta string
  const titleMatch = meta?.match(/title="([^"]*)"/)
  const title = titleMatch?.[1]

  const html = await codeToHtml(code, { ... })

  // Pass title to client component
  return <CodeBlockClient html={html} code={code} title={title} />
```

**`components/docs/mdx/code-block-client.tsx`** — Replace with the latest template version. Adds title bar rendering and adjusted button positioning.

**`components/docs/mdx/image-lightbox.tsx`** — Replace with the latest template version. Adds width control via the markdown title attribute.

**All files containing `MDXRemote`** — Add `rehypeCodeMeta` to the rehype plugins array:

```ts title="Any file with MDXRemote"
import { rehypeCodeMeta } from '@/lib/rehype-code-meta'

// In MDXRemote options:
rehypePlugins: [rehypeSlug, rehypeCodeMeta],
```

This applies to: `app/(docs)/[...slug]/page.tsx`, both blog layout article components, and both release notes layout article components.

### Step 1: Remove deprecated dependencies

```bash title="Terminal"
npm uninstall redoc styled-components mobx
```

If you use the Redoc-based API reference pages, keep these dependencies. The template no longer includes them, but they still work if you need them.

- Tab elements excluded from prose list markers
- Code block margins reduced

### Step 3: Fix known template issues

If you scaffolded before v1.5.0, check for these issues:

- **Empty `app/(docs)/api/[...slug]/` directory** — Delete it if present and you're not using API docs

- **Heading hydration mismatch** — Update `components/docs/mdx/heading.tsx` to use `useState` + `useEffect` pattern from the latest template

## Troubleshooting

### `lightningcss` or `@tailwindcss/oxide` errors on Windows

If your `.npmrc` contains `os = "linux"`, npm will skip Windows-specific optional dependencies. Fix by running:

```bash title="Terminal"
npm install --os=win32
```

Or add the Windows binaries to your `package.json`:

```json title="package.json"
{
  "optionalDependencies": {
    "@tailwindcss/oxide-win32-x64-msvc": "^4.2.0",
    "lightningcss-win32-x64-msvc": "^1.31.1"
  }
}
```

### `.next/trace` EPERM error during build

On Windows, a stale Node.js process may hold a lock on the `.next` directory. Kill all Node processes and clean the build cache:

```bash title="Terminal"
taskkill /f /im node.exe
rm -rf .next
npm run build
```

### `vars is not defined` in MDX

Make sure the `MDXRemote` options include the `scope` property:

```tsx title="MDXRemote usage"
<MDXRemote
  source={content}
  components={mdxComponents}
  options={{
    scope: { vars: docVariables },
    mdxOptions: { ... },
  }}
/>
```

### Build fails with "Could not parse expression with acorn"

This occurs if `rehype-mdx-code-props` is installed — it tries to parse code block meta strings like `{2,4-5}` as JavaScript expressions. Remove it and use the custom `rehypeCodeMeta` plugin from `lib/rehype-code-meta.ts` instead.
