{vars.productName} includes a migration script that automates the bulk of moving from Docusaurus. It copies your content, converts your sidebar, transforms frontmatter, and flags anything that needs manual attention.

## Prerequisites

Before running the migration:

1. **Scaffold a {vars.productName} project** â€” follow [Getting Started](/getting-started/) to create a new project
2. **Have your Docusaurus project accessible** â€” the script reads from its `docs/` directory
3. **Back up your Docusaurus project** â€” the script only reads from it, but backups are always a good idea

## Running the migration

```bash
node scripts/migrate-docusaurus.js <path-to-docusaurus-project>
```

### Options

| Flag | Description |
|------|-------------|
| `--dry-run` | Preview what would change without writing any files |
| `--force` | Overwrite existing files in `content/docs/` |

Always run with `--dry-run` first to review the changes:

```bash
node scripts/migrate-docusaurus.js ../my-docusaurus-site --dry-run
```

Then run without the flag to perform the migration:

```bash
node scripts/migrate-docusaurus.js ../my-docusaurus-site
```

## What the script does

The migration runs in three phases.

### Phase 1: Content files

The script copies every `.md` and `.mdx` file from the Docusaurus `docs/` directory into `content/docs/`, applying these transforms:

#### File paths

| Transform | Example |
|-----------|---------|
| Numbered prefixes stripped | `01-getting-started.md` â†’ `getting-started.mdx` |
| `.md` renamed to `.mdx` | `guide.md` â†’ `guide.mdx` |
| `README.md` renamed to `index.mdx` | `api/README.md` â†’ `api/index.mdx` |

#### Frontmatter

Docusaurus-only frontmatter fields are removed automatically:

- `sidebar_position`, `sidebar_label`, `displayed_sidebar`
- `id`, `slug`, `pagination_label`, `pagination_next`, `pagination_prev`
- `custom_edit_url`, `parse_number_prefixes`, `hide_title`

The `tags` field is converted to `keywords` (the Trellis equivalent).

:::caution
If your Docusaurus pages use custom `slug` values, the migration report will warn you. URLs may change after migration since {vars.productName} derives routes from file paths.
:::

#### Content body

| Transform | Details |
|-----------|---------|
| `@theme` imports removed | `import Tabs from '@theme/Tabs'` â€” Trellis provides these automatically |
| `@site` imports removed | Warns you to verify no custom components are missing |
| HTML comments converted | `<!-- comment -->` â†’ `{/* comment */}` (MDX syntax) |
| MDX partial imports converted | `import Foo from './_foo.mdx'; <Foo />` â†’ `@include ./_foo.mdx` |
| Excessive blank lines cleaned | Three or more blank lines reduced to two |

The script warns about `<CodeBlock>` components and non-Docusaurus imports that need manual review.

#### Assets

Non-markdown files (images, PDFs, etc.) in the `docs/` directory are copied alongside the content with the same path transforms.

### Phase 2: Sidebar conversion

The script reads your Docusaurus sidebar configuration and generates a {vars.productName} `config/sidebar.ts` file.

| Docusaurus source | How it's handled |
|-------------------|-----------------|
| `sidebars.js` / `.cjs` / `.json` | Loaded and converted directly |
| `sidebars.ts` | Parsed as plain data (TypeScript types stripped) |
| `type: 'autogenerated'` | Generated from the filesystem after content is copied |
| `type: 'link'` (external) | Skipped with a warning |
| `_category_.json` labels | Used for category names in the generated sidebar |
| No sidebar file found | Generated entirely from the filesystem |

If a `config/sidebar.ts` already exists, the script backs it up to `config/sidebar.backup.ts` before writing.

### Phase 3: Variable suggestions

After migration, the script scans your content for repeated strings â€” version numbers, product names, and URLs that appear three or more times. These are candidates for [content variables](/guides/site-configuration/#content-variables) in `config/variables.ts`.

## After migration

### 1. Review the sidebar

Open `config/sidebar.ts` and verify the structure matches your expectations. You may want to:

- Reorder items
- Adjust `collapsed` values
- Add `link` properties to categories that have index pages

See [Configuring the sidebar](/guides/content-authoring/#configuring-the-sidebar) for the full sidebar API.

### 2. Run the build

```bash
npm run build
```

Fix any MDX compilation errors. Common issues:

| Error | Fix |
|-------|-----|
| JSX expression expected | HTML comments missed by the converter â€” change `<!-- -->` to `{/* */}` |
| Unknown component | A Docusaurus component (`<CodeBlock>`, `<BrowserWindow>`) that needs to be removed or replaced |
| Expression expected | Bare `<` or `>` characters in text â€” wrap in backticks or escape as `&lt;` / `&gt;` |

### 3. Update site configuration

Edit `config/site.ts` with your branding, logo, and URLs. See [Site Configuration](/guides/site-configuration/) for all options.

### 4. Add variables

If the migration report suggested variables, add them to `config/variables.ts` and replace hardcoded values in your content with `{vars.key}`:

```ts title="config/variables.ts"
export const docVariables: Record<string, string> = {
  productName: 'Acme Platform',
  version: '2.1.0',
}
```

### 5. Set up navigation

Edit `config/navigation.ts` to configure your top navbar and footer. See [Navigation](/guides/site-configuration/#navigation).

### 6. Add redirects

If your Docusaurus site used custom slugs or numbered prefixes, URLs will have changed. Add entries to `redirects.json` so old links still work:

```json
[
  { "from": "/api-reference/", "to": "/guides/api/" },
  { "from": "/01-intro/", "to": "/intro/" }
]
```

See the [Redirects plugin](/plugins/redirects/) for details.

## What the script does not migrate

| Item | Why | What to do |
|------|-----|------------|
| Blog posts | Different directory structure | Copy from `blog/` to `content/blog/` manually |
| Custom pages | Docusaurus uses React pages in `src/pages/` | Recreate in `app/` as Next.js pages |
| Custom CSS | Docusaurus uses `src/css/custom.css` | Map to [design tokens](/design-tokens/customizing/) |
| Plugins | Docusaurus plugins have no Trellis equivalent | Use built-in features or build custom components |
| Versioned docs | Different snapshot format | Re-snapshot with `npm run version:snapshot` after migration |
| i18n translations | Different directory structure | Copy translated content into `content/i18n/<locale>/docs/` |

## Syntax compatibility

Most Docusaurus markdown syntax works in {vars.productName} without changes:

| Feature | Docusaurus | {vars.productName} | Status |
|---------|-----------|------------|--------|
| Admonitions | `:::tip` | `:::tip` | Works as-is |
| Custom admonition titles | `:::tip Custom Title` | `:::tip Custom Title` | Works as-is |
| Tabs | `<Tabs>` / `<TabItem>` | `<Tabs>` / `<TabItem>` | Works â€” no import needed |
| Code blocks | ` ``` ` | ` ``` ` | Works as-is |
| Frontmatter | YAML | YAML | Works â€” Docusaurus fields stripped |
| MDX comments | `{/* comment */}` | `{/* comment */}` | Works as-is |
| HTML comments | `<!-- comment -->` | Not supported | Auto-converted by script |
| Partials (import) | `import Foo from './_foo.mdx'` | `@include ./_foo.mdx` | Auto-converted by script |
| `<DocCardList />` | `<DocCardList />` | `<DocCardList category="..." />` | Same name â€” pass `category` prop |
| Image width | Not supported | `![alt](/img/pic.png "50%")` | New â€” set width via title attribute |
| Variables | Not supported | `{vars.key}` | New feature |
