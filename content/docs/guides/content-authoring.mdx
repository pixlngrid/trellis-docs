This guide covers how to create and organize content in {vars.productName}. For markdown syntax and features (code blocks, admonitions, tabs), see [Writing Docs](/guides/writing-docs/).

## Creating a New Page

Add an `.mdx` file under `docs/`. The file path determines its URL:

| File Path | URL |
|-----------|-----|
| `docs/getting-started.mdx` | `/getting-started/` |
| `docs/guides/deployment.mdx` | `/guides/deployment/` |
| `docs/theme/index.mdx` | `/theme/` |

Use `index.mdx` for the landing page of a section (e.g. `theme/index.mdx` → `/theme/`).

### Partial Files

Files and folders that start with an underscore (`_`) are treated as **partials** — they are excluded from routing, search indexing, and the sidebar. This convention matches [Docusaurus behavior](https://docusaurus.io/docs/creating-pages#routing), so content that worked in Docusaurus carries over.

```
docs/
├── guides/
│   ├── deployment.mdx            → /guides/deployment/
│   └── _internal-notes.mdx       ✗ no route, not indexed
├── _shared/
│   └── common-setup.mdx          ✗ entire folder skipped
└── getting-started.mdx           → /getting-started/
```

Use partials for:

- **Internal reference notes** that shouldn't be published
- **Draft content** that isn't ready for readers yet
- **Reusable snippets** you plan to incorporate into other pages

To hide a file, simply prefix its name with `_`:

```bash
# Hide a page
mv docs/guides/draft.mdx docs/guides/_draft.mdx

# Hide an entire folder
mv docs/internal docs/_internal
```

Partial files are also excluded from the [Smart Search](/plugins/smart-search/) index via the `excludedPrefixes` option.

### Frontmatter

Every page starts with YAML frontmatter:

```yaml
---
title: My Page Title
description: A short description for search results and SEO.
keywords: [keyword-one, keyword-two]
last_update:
  date: 02/22/2026
  author: Your Name
doc_type: how-to
role: [Developer, Admin]
draft: false
---
```

| Field | Required | Description |
|-------|----------|-------------|
| `title` | Yes | Page title, displayed as the h1 heading |
| `description` | No | Used in search results and meta tags |
| `keywords` | No | Array of keywords to boost search relevance |
| `last_update.date` | No | Displayed at the top of the page (MM/DD/YYYY) |
| `last_update.author` | No | Subject matter expert for the page (visible when `lastUpdated.showAuthor` is `true` in `config/site.ts`) |
| `hide_table_of_contents` | No | Set to `true` to hide the right-side TOC |
| `doc_type` | No | Content type for auditing (e.g. `how-to`, `concept`, `reference`, `tutorial`). Not rendered on the page — used in sidebar exports |
| `role` | No | Array of audience roles (e.g. `[Developer, Admin]`). Rendered as colored chips below the page title |
| `draft` | No | Set to `true` to exclude the page from the production build, search index, and FAQ index |

### Role Chips

When you add a `role` array to a page's frontmatter, Trellis renders each role as a colored chip below the title. This helps readers quickly identify whether a page is relevant to them.

```yaml
---
title: API Authentication
role: [Developer, Admin]
---
```

Roles are purely informational — they don't restrict access. Use any labels that make sense for your audience (e.g. `Developer`, `Admin`, `Designer`, `Product Manager`).

### Document Type

The `doc_type` field is metadata-only — it is not rendered on the page. Use it to classify pages for content auditing:

| `doc_type` | When to use |
|------------|-------------|
| `concept` | Explains what something is or why it matters |
| `how-to` | Step-by-step instructions for a specific task |
| `reference` | API docs, config options, parameter tables |
| `tutorial` | End-to-end guided walkthrough |

Both `doc_type` and `role` are included in the [sidebar export](#sidebar-export) for spreadsheet-based auditing.

### Draft Pages

Set `draft: true` in a page's frontmatter to exclude it from the production build:

```yaml
---
title: Upcoming API Changes
draft: true
---
```

Draft pages are:

- **Excluded from the build** — no static page is generated, so the URL returns a 404 in production
- **Excluded from search** — not indexed by the [Smart Search](/plugins/smart-search/) plugin
- **Excluded from FAQ index** — not indexed by the [FAQ Index](/plugins/faq-index/) plugin
- **Visible in sidebar exports** — the [sidebar export](#sidebar-export) includes a Draft column so you can track which pages are still in progress

:::tip
Use `draft: true` for pages that are actively being written. Unlike the `_` prefix convention (which hides files from the filesystem level), drafts stay in the sidebar config and appear in content audits — making them easier to track.
:::

## Configuring the Sidebar

The sidebar is defined in `config/sidebar.ts`. If you've used Docusaurus, this plays the same role as `sidebars.js` — but it's a typed TypeScript array instead of a plain object.

### Item Types

There are two item types: **doc** (a link to a page) and **category** (a collapsible group).

```ts
export const mainSidebar: SidebarItem[] = [
  // A standalone doc at the top level
  { type: 'doc', id: 'getting-started' },

  // A category with child docs
  {
    type: 'category',
    label: 'Guides',
    items: [
      { type: 'doc', id: 'guides/content-authoring' },
      { type: 'doc', id: 'guides/deployment' },
    ],
  },
]
```

### Doc Items

A `doc` item links to a page. The `id` is the file path under `docs/` without the extension:

| File | `id` |
|------|------|
| `docs/getting-started.mdx` | `'getting-started'` |
| `docs/guides/deployment.mdx` | `'guides/deployment'` |
| `docs/theme/index.mdx` | `'theme/index'` |

The sidebar label defaults to a title derived from the filename. Override it with the `label` property:

```ts
{ type: 'doc', id: 'overview/index', label: 'What is Trellis?' }
```

### Categories

A `category` is a collapsible group with a label and child items:

```ts
{
  type: 'category',
  label: 'Guides',
  collapsed: true,
  items: [
    { type: 'doc', id: 'guides/content-authoring' },
    { type: 'doc', id: 'guides/writing-docs' },
  ],
}
```

| Property | Required | Description |
|----------|----------|-------------|
| `label` | Yes | Display text in the sidebar |
| `items` | Yes | Array of child `doc` or `category` items |
| `collapsed` | No | Start collapsed (`true`) or expanded (`false`, default) |
| `link` | No | Doc id to make the category label itself a clickable link |

### Linking a Category to a Page

By default, clicking a category label just toggles it open/closed. Use the `link` property to make the label navigate to a page — the chevron still toggles expand/collapse independently:

```ts
{
  type: 'category',
  label: 'Theme',
  link: 'theme/index',    // clicking "Theme" navigates to /theme/
  collapsed: true,
  items: [
    { type: 'doc', id: 'theme/last-updated' },
    { type: 'doc', id: 'theme/tabs' },
  ],
}
```

This avoids the need for a redundant child like "Theme Overview" under a "Theme" category. Categories without `link` (like "Guides") remain plain collapsible headers.

### Nesting Categories

Categories can be nested for deeper hierarchies:

```ts
{
  type: 'category',
  label: 'API',
  items: [
    {
      type: 'category',
      label: 'REST',
      items: [
        { type: 'doc', id: 'api/rest/authentication' },
        { type: 'doc', id: 'api/rest/endpoints' },
      ],
    },
    { type: 'doc', id: 'api/graphql' },
  ],
}
```

## Landing Page

By default, the root URL (`/`) renders a landing page with a hero section and feature cards. This is defined in `app/page.tsx`.

### Skipping the Landing Page

Most documentation sites don't need a marketing-style landing page — visitors should go straight to the docs. To bypass the landing page and redirect `/` to your first doc, replace the contents of `app/page.tsx`:

```tsx title="app/page.tsx"
import { redirect } from 'next/navigation'

export default function Home() {
  redirect('/getting-started/')
}
```

Change `'/getting-started/'` to whichever page you want as your entry point. The redirect works with static export — Next.js generates an HTML page with a `<meta http-equiv="refresh">` tag, so no server is required.

:::tip
After switching to a redirect you can also remove imports and assets that were only used by the landing page, such as the hero logo in `config/site.ts` (`logo.hero`) and any feature card data.
:::

### Customizing the Landing Page

If you want to keep the landing page but tailor it, edit `app/page.tsx` directly. It's a regular React component — not MDX — so you have full control over layout. The key pieces to update are:

- **Hero text** — the tagline and call-to-action buttons
- **Feature cards** — the `features` array at the top of the file
- **Hero logo** — controlled by `logo.hero` and `logo.useBuiltIn` in `config/site.ts`

## Images

Place images in `public/img/` and reference them with an absolute path:

```md
![Architecture diagram](/img/architecture.png)
```

All images automatically support **click-to-zoom** via the built-in lightbox — no extra markup needed. Click any image to open it in a modal overlay.

## Blog Posts

Blog posts live in `blog/`. Name files with a date prefix:

```
blog/2026-02-20-welcome.md
```

### Blog Frontmatter

```yaml
---
title: Welcome to Trellis
description: Introducing the Trellis docs framework.
slug: welcome
authors:
  - name: Trellis Team
---
```

| Field | Description |
|-------|-------------|
| `title` | Post title |
| `description` | Short description |
| `slug` | URL slug (defaults to the filename slug) |
| `authors` | Array of `{ name }` objects |

The date is parsed from the filename. Posts are listed at `/blog/`, newest first.

### Excerpt Truncation

Use an MDX comment to mark where the excerpt ends on the blog index page:

```mdx
This intro paragraph appears on the index page.

{/* truncate */}

This content only appears on the full post page.
```

## Release Notes

Release notes live in `release-notes/`. Name files after the version:

```
release-notes/v1.0.0.md
```

### Release Notes Frontmatter

```yaml
---
title: v1.0.0
version: 1.0.0
date: 2026-02-20
description: Initial release of Trellis on Next.js.
---

## Added
- Smart search with configurable weights
- Design token pipeline

## Changed
- Migrated from Docusaurus to Next.js

## Removed
- Legacy search plugin
```

| Field | Description |
|-------|-------------|
| `title` | Display title |
| `version` | Semantic version for sorting |
| `date` | Release date (YYYY-MM-DD) |
| `description` | Short summary |

Release notes are listed at `/release-notes/`, sorted by version (newest first).

## Variables

{vars.productName} supports **reusable variables** in MDX files — a feature Docusaurus doesn't offer. Define variables once, use them across every page.

### Defining Variables

Edit `config/variables.ts` to add or change variables:

```ts title="config/variables.ts"
export const docVariables: Record<string, string> = {
  productName: 'Trellis',
  companyName: "Pixl'n Grid",
  version: '1.0.0',
  frameworkName: 'Next.js',
}
```

### Using Variables in MDX

Reference any variable in your content with `{vars.variableName}`:

```mdx
Welcome to {vars.productName} by {vars.companyName}.

{vars.productName} is built on {vars.frameworkName} and is currently
at version {vars.version}.
```

This renders as:

> Welcome to {vars.productName} by {vars.companyName}.
>
> {vars.productName} is built on {vars.frameworkName} and is currently at version {vars.version}.

Variables work anywhere in MDX — paragraphs, headings, tables, admonitions, and JSX component props.

:::tip
When you rename a product or bump a version, update `config/variables.ts` once and every page reflects the change on the next build.
:::

## Sidebar Export

Trellis includes a script that exports your sidebar structure as a CSV file for content auditing in spreadsheets. Run it with:

```bash
npm run export:sidebar > sidebar.csv
```

The CSV includes one row per sidebar entry with these columns:

| Column | Source |
|--------|--------|
| Doc ID | The sidebar `id` (e.g. `guides/content-authoring`) |
| Title | Sidebar label or frontmatter `title` |
| URL | The page URL |
| Category | Sidebar category path (e.g. `Guides > Content Authoring`) |
| Draft | `Yes` if `draft: true`, blank otherwise |
| Doc Type | Frontmatter `doc_type` |
| Role | Frontmatter `role` (semicolon-separated) |
| Last Updated | Frontmatter `last_update.date` |
| Author | Frontmatter `last_update.author` |
| Description | Frontmatter `description` |
| Keywords | Frontmatter `keywords` (semicolon-separated) |
| File Path | Relative file path from project root |

:::tip
Import the CSV into Google Sheets or Excel to audit coverage gaps, identify stale pages, and track which roles are documented.
:::

## MDX Tips

- **Comments** must use `{/* comment */}` — HTML comments (`<!-- -->`) cause build errors in MDX.
- **Components** are available globally in MDX without importing. See [Using Components](/guides/using-components/) for the full list.
- **Blank lines** are required before and after fenced code blocks and JSX components for correct parsing.
